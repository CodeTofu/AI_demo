# 前端代码审查

## 何时使用

适用于 `frontend/`（或任意浏览器 UI）下的变更：组件、页面、hooks、API 客户端、样式、路由，以及影响打包体积的构建配置。

## 审查流程

1. **范围**：理清涉及的路由、共享组件、全局 CSS 等。
2. **上下文**：打开相关文件，对齐既有写法（导入、状态、样式）。
3. **输出**：使用下文严重级别；尽量给出文件路径与行号范围。

## 检查清单

### 正确性与数据流

- [ ] Props/状态与用法一致；除非有意为之，避免 effect 中的陈旧闭包。
- [ ] 异步流程覆盖加载、错误、空状态；必要时取消或忽略过期响应。
- [ ] 表单：受控输入、校验与提交反馈与后端/API 约定一致。

### TypeScript 与可维护性

- [ ] 类型与运行时数据一致；避免滥用 `any`，若使用需有理由且范围小。
- [ ] 共享结构放在公共类型/模块；孤立场景下的重复可接受。
- [ ] 组件职责集中；仅在能消除真实重复时再拆分大文件。

### UI、体验与样式

- [ ] 常见断点下布局正常；无明显溢出或焦点环被裁切。
- [ ] 文案、加载与错误提示清晰，且与产品整体一致。
- [ ] CSS 符合项目约定（变量、类名、同目录 `.css` 等）。

### 无障碍（a11y）

- [ ] 可交互元素可用键盘到达；焦点顺序合理。
- [ ] 有语义的图片/图标提供文本替代。
- [ ] 状态不只靠颜色区分；文字与控件的对比度合理。

### 性能

- [ ] 列表避免无谓整表重渲染；`key` 稳定。
- [ ] 重活不放每次渲染；高频事件酌情防抖/节流。
- [ ] 大依赖或大资源有理由；明显受益时再懒加载路由或分包。

### 安全（浏览器侧）

- [ ] 未经消毒不使用 `dangerouslySetInnerHTML`；用户可控字符串需转义或校验。
- [ ] 不在日志或客户端包里暴露 token/密钥。
- [ ] 涉及安全时，对用户输入产生的 URL/跳转做校验或白名单。

### API 与环境

- [ ] 基础 URL 与环境变量用法与后端一致；`fetch`/客户端错误能向用户说清楚。

## 反馈格式

单次审查内保持标签一致，建议使用：

- **必须修**：正确性、安全或严重体验问题；阻塞合并。
- **建议**：有明确收益的改进；不阻塞。
- **小问题**：风格、命名或可选润色。

末尾写 **总结**（2～4 句）：整体风险、主要问题类型、是否建议在修改意见后通过。

## 与项目对齐

优先沿用仓库现有技术栈与风格（如 React + TS + 同目录 CSS）。除非变更已引入或用户明确要求，不要擅自推荐新库。

## 可选深入

当改动涉及焦点、ARIA 或读屏行为时，再阅读 [a11y-notes.md](a11y-notes.md) 中的补充说明。
